Loom SDK API Documentation¶

The target audience for this section of Loom documentation are the Loom App developers and 3rd party developers trying to build Enterprise Information Management Applications that run on Loom.

This is programmer’s SDK Guide.

For basic concepts on Loom, refer to Loom: Concepts & Glossary documentation.

Loom is comprised of several information management micro-services which can be used for several information management functionalities such as filtering data, scanning data, classifying data, gaining visibility into what kind of enterprise data is stored across enterprise data assets and more. The micro-service based Loom architecture is built from ground up to facilitate development of Data Management Applications running on top of the core Loom Platform. One such application is in-built with Loom Alpha1 Deployment - 360 Data Analysis. This application addresses enterprise needs for data management such as Visibility, Classification and capability to securely connect to all enterprise data sources irrespective of wherever they are - in the cloud or on premise. In future, there may be additional in-built applications with core Loom services that will be part of Loom deployment. For e.g., application to address enterprise GDPR use case.

Post Alpha1 release, Loom will offer enterprises the capability to develop 3rd party data management applications and deploy them on the Loom ecosystem. This can be done by using the REST based Loom APIs that will be part of the future releases.

This document lists Loom SDK APIs as a developer reference. It will be refined and populated as Loom evolves beyond Alpha1 release. As of now, this document is organized as follows:

For each service, there is a dedicated section covering basic overview of the service, how it can be used and for what purpose. This is followed by Service API listing as per the service .json file. Next, there is a FAQ document covering some of the frequently asked questions about the service usage. In some cases you may find a service troubleshooting guide and additional information. For e.g., in case of Identity and Access Management service, there is additional guideline and best practice document covering password rules.

Overview of Loom Platform UI Service¶

The Platform UI Service of Loom Platform provides interfaces that help in metadata manipulation and analyzing data managed by Loom Plaform. This data manipulation at the front end by Loom Applications involves adding additional useful fields, removing fields to enhance security and showing views of the data based on user role and access permissions. For example - these APIs can help user in listing all the applications that the user has access to and are deployed by Loom..

There can be multiple kinds of Loom Applications:

  • Non UI Applications - These applications could simply invoke Loom UI Service or other public REST APIs exposed by the Loom Platform programmatically.
  • Web based Applications - These applications could utilize the Web browser to run Loom REST API interfaces - Platform UI or other public Loom REST APIs.
  • Angular based Applications - Angular-based applications are encouraged to use the functionality provided by the shared library instead of directly using most of these REST API calls.

Using Platform UI Service¶

All applications can access platform services using the rest API directly. For details, refer to Getting Started with Platform UI Service, Platform UI Service: APIs, Platform UI Service : FAQ and Platform UI Service: Troubleshooting Guide.

Loom Application Information Files¶

Every application using the shared angular module should host an application information file at a standard location and its locale files also at a standard location. This metadata provides the platform (and other applications) with descriptive information and understanding about your application’s capabilities.

These files extend the information (display name, icon and other fields) that the application manager provides.

Creating an application info file¶

If you are developing a Loom Application, your application’s information is stored in a JSON format file. The file is read by the Loom Platform by downloading it from ‘http://your-applications-url/ext/application-info.json’. So, it needs to be placed at the root of your application web server with the correct file name. This file informs the platform of your application name, navigation links and supported locales. Other items may be added in the future.

You are encouraged to use RBAC capabilities if possible in the navigation section. The navRoles attribute is deprecated expect for situations like the System Level Role Permissions that aren’t covered by RBAC today.

Loom Policy Management Service¶

Getting Started with Policy Management¶

Loom platform hosts Veritas or third party applications which provide different capabilities and actions ( Discovery, Scan, Classify etc ) to the users. Loom Policy Management provides a common framework for users of Loom applications to define and control application behavior. This application behavior is nothing but actions executed by applications on data sources or assets. The users can use Policy Management framework to define access and action policies across all data sources and for all applications. This helps users to bring in enterprise specific business rules for data analysis, scanning and classification etc. Policy Framework supports multi-tenancy where policy definition and enforcement happens in the context of tenant.

Overview¶

Note

Version

Description

Policy Management Framework is a set of core platform services which help users to define policies which triggers Actions ( such as SCAN, DISCOVER, DELETE etc ). A Policy describes when the action will occur and scope of the action ( set of rules ). The Action triggered by a policy is executed by application responsible for that action. E.g. Classification policy triggers Classify action which is executed by Classification Service ( which is part of Data visibility application).

Policy management framework is composed of following components:

  1. Policy Administration : Provides API for defining policy and triggering the execution.
  2. Policy Store: A multi-tenant repository of all authored policy
  3. Policy Dashboard: Provides a user interface to author polices and trigger actions. Policy dashboard interacts Policy Administration service in the background.
  4. Policy Execution Service: Interacts with Job Management and Asset Service to trigger actions.
  5. Policy Auditing: Responsible for auditing the policy related events.

Consumers of this service are Platform Admin and Tenant Admin. [TBD]

Policy registration¶

Policies defined in the framework are in the context of application. When application is installed on Loom platform, it declares the type of policies it can support and some pre-defined policies corresponding to the type. This process is called registration. This way a new application can dynamically add new capabilities along with policies.

Managing Policies¶

Policy Framework provide API to perform create, edit and delete operations on policy. Refer to API documentation for more details. Each of this operation affects policy definition in the policy database.

Each Policy has following attributes:

Attribute Purpose
Name Unique policy name per tenant
Type SCAN/DISCOVER/CLASSIFICATION/DELETE/EXPORT
Description User defined description
Scope/RuleSets User can create multiple RuleSets in one policy. Each RuleSet is combination of Various rules of type inclusion/exclusions/filters/generic. The rules that determine the scope of the policy e.g. All PDF files owned by users in user-group A. This is opaque to the policy service framework and is interpreted and evaluated by the specific service responsible for running the action once the action is triggered
When/Schedule Can be scheduled for a specific time (once or recurring)

Publish Policy or Binding Policy¶

Create policy is just a policy definition and does not trigger any Action. For action to be triggered, policy instance needs to be created by binding policy definition to assets and schedule. This operation is referred as bind or publish or somethings job creation.

During bind operation, Policy Execution Service creates a job definition with Job Management Service. Job management schedules the job as per schedule ( or immediately if there is no schedule defined). This job is picked up by application for execution

  1. User lands on Policy Management screen

    • Create Schedule using Schedule options provided
  4. User can also choose, publish now to schedule jobs immmediately
  5. User choose to publish by clicking publish button

General¶

What is Loom Policy Service? Loom Policy Management framework comprises of a common policy language for defining, aggregating, decomposing, executing policies and reporting policy outcomes. It includes a policy definition component for domain specific policies, policy enforcement engine for triggering actions and auditing engine for audit and compliance. The Loom Policy Management functionality helps to bring in enterprise specific business rules for data analysis, scanning and classification.

What is Loom Policy? Policy is a set of of rules that control application behavior. It is a statement of intent, and can be implemented as

  • Ability - Is this actor permitted to perform this action ?
  • Threshold - If something exceeds certain pre-defined threshold, then take an action (like running low on storage, add block devices to LVM group)
  • Procedure - Perform action based on business logic
  • Protocol - Do an action due to regulation or compliance

Policies can be of many types like Availability, Performance, Mobility, Protection, Retention, Cost, Security and Privacy.

Policy is core to all Veritas products and all implement policy management in different ways. Some policies are Request driven. i.e. business policy driven to get desired behavior. Some others are Event driven i.e. when a certain event happens, then perform some action

What are the key components of Loom Policy Management?

Loom Policy Service comprises of the following components:

  • Policy Database - a repository of all authored policies
  • Policy Dashboard - A UI through which users can define policies. These policies are validated and stored in the policy database.
  • Policy Administration - The entity that maintains the complete policy model for all policies. The policy server can decompose policies into substituents or can aggregate different policies, depending on the nature of policies defined. It acts as Policy Decision Points (PDP).
  • Policy Engine - Policy engine is responsible for enforcement of policies (retrieved from policy store). The enforcement to the decision is with policy engines with product specific implementations. Policy engine is a Policy Enforcement Point (PEP).
  • Policy Audit Store - For auditing policy related events in the system.

How do I get started with Loom Policy Management Service?

Policy Dashboard is a place for authoring, enforcing and monitoring. Refer to getting started guide.

Where can I see all the policies setup for a typical Loom deployment?

Each application’s dashboard has policies tab from where user can view policies and take actions (Edit, Delete, Publish )

Are there any default policies that are present in a typical Loom Deployment?

Yes. During application install, application can register its own policy types representing supported actions, connectors and assets. Application can also provide some pre-defined policies to policy framework. One of these pre-defined policies can be marked as default. A default policy has a default schedule and asset type. If the Asset corresponding to this type gets added in the system, default policy is automatically applied.

Is Policy Management Multi-tenant?

Yes. Both policy definition and enforcement happens in the context of tenant.

Are there any other Loom Services that Policy Management is dependent upon?

Yes. These services are: IDM, RBAC and Asset Service, Job Management. More to be updated in time. TBD.

Other General Info about Policy Service

Policy Framework is agnostic of grammar of policies defined by the application. Neither it is responsible for validating the policy grammar. Applications expose validation hook ( Rest endpoint ) for validation, on create policy call, policy manager will call it to validate policy grammar.

Application Manager Service¶

Application Manager is a platform REST service that manages the lifecycle of an application. Application lifecycle entails the following activities:

  • Installation of a Loom application
    • Bring up pods
    • Add schemas into the database (optional)
    • Generate certificates for the application
    • Register the application with IDM
    • Creation of topics on Kafka (optional)
    • Creation of log indexes, metrics store etc
  • Application Upgrades
    • Update schemas and perform database migration (optional)
    • Perform a canary upgrade of the pods
  • Delete an application
    • Graceful stop of pods and deletion
    • Deleting of schemas in database (optional)
    • Revoking certificates from CA
    • De-register application from IDM
    • Deletion of topics from Kafka (optional)
    • Cleanup logs, metrics, alerts from the application

The application manager will provide REST APIs to manage application lifecyle. In addtion, it will also provide APIs to query application of their endpoints,versions and associated resources. Application Manager may provide a UI to faciliate the onboarding of an application (like an installation wizard) in addition to the REST API.

A high level interaction diagram is shown below. The application manager will be responsible for installing the application artifacts and configuring different components for the new application (eg: Elasticsearch index, database schema, certification generation etc.).

Figure: Loom Application Manager Architecture

During Installation, application manager extracts application manifest from installer package and persist application properties in database. An application comprises of one OR more components. Application installation package contains a manifest file (template.json) which defines application and meta data of all components (name, version). It also contains details of application’s docker registry from where images needs to be pulled. Installer package also contains helm charts of application’s components. Each component’s helm chart contains a manifest file (loomspec.yaml) which describes that particular component.

Application manager invokes airflow DAG, for application installation and airflow then internally uses rudder for actual application installation. For each component (as defined in template.json) airflow invokes rudder’s api to install release. Rudder, while installation pulls application’s docker images, from configured docker registry and brings up pods.

Once install request is submitted to airflow, application manager return status (202 accepted) to the client. Airflow then invokes application manager’s PUT request (/apps/updateState) for updating status of ongoing installation. With each update request, app-manager will update application/component state is DB to show real time progress.

Applications Manifest¶

Each Loom Application is identified by an Application Manifest file. This file defines each application in terms of key Loom Platform micro-services that are required to run this application and other resources such as what kind of jobs are associated with this application, the kind of role based access control needed for application workflows and the kind of Loom Policies associated with this application.

The Application manifest is a directory consisting of the following files that define the application’s components and supported resource types or ‘kinds’.

Applicationspec.json - list various constituents of a Loom application. JobKindSpec.yaml - defines the JobType type that can be used by Loom services to define different Job types RBACKindSpec.yaml - defines the RBACRuleType type that can be used by Loom services to define different RBAC Rule types PolicyKindSpec.yaml - defines the PolicyType type that can be used by Loom services to define different Policy types

What is Application Manager Service?¶

Application manager service is responsible for managing life-cycle of application on loom platform. Platform admin (via app-mgr UI) can use them for installing, uninstalling, upgrading applications on platform.

Service Dependencies¶

Airflow is needed to run install, uninstall, upgrade DAGs. Rudder is needed for helm chart install/uninstall/upgrades. Postgres DB for persisting application details.

When to use this service?¶

Platform admin will use this service (via UI) for installing/uninstalling/upgrading applications, as and when needed.

Service Usage REST API Workflow¶

From UI, admin will upload application installer package. Application manager will extract application & components manifest file from installer package and persist application’s properties in database. Application manager will invoke airflow DAG, which internally invokes rudder api for installing application. Airflow will update install progress to application manager via updateState api.

The application not only has access to the platform services and infrastructure, but also shares the resources, the security domain and the multi-cloud connectivity. The platform itself is treated like an application. For more details see Loom Application.

General¶

  1. What is Loom Identity and Access Management?

    You can use Loom IDM to securely control individual and group access to the most critical enterprise resource – information such as location of enterprise data, its attributes such as highly shared data, orphan files, sensitive information, data access history obtained from enterprise data assets filers, cloud storage, and other secondary storage including backup and archives.

  2. How do I get started with IDM?

    After you have signed up or registered for Loom (hosted SaaS model) or deployed Loom on premise or on cloud within your enterprise, you can create users and groups and assign them permissions to access Loom information resources and applications that can help in gaining dynamic insights about enterprise data. You can use IDM for web-based access to the Loom Dashboard and its user interfaces, or the API or SDK (for programmatic access). In order to grant permissions, you need to do X, Y, Z. For details see – Getting Started with Loom IDM (link dev document here).

  3. What end user pain points or problems does IDM solve?

    IDM makes it easy to provide multiple users secure access for enterprise information governance related activities that access the most critical enterprise resource – data assets. IDM enables you to:

    • Manage Loom users and their access
    • Control who has access to what information insights within an organization

  4. Who can use IDM?

    Any Loom user can use IDM. This is a core service that is offered at no additional charge or billing.

  5. What is a user?

    A user is a unique identity recognized by Loom services and applications, belonging to a customer account. Users can be grouped into multiple tenants. Each customer account can have more than one tenant and every tenant is associated with one or more users. A user can belong to more than one tenant but Loom internally maintains unique user IDs that are linked together for such users. (Need to double check on this one). Similar to Operating Systems, a user has a unique name and can identify itself using security credentials such as password or access key. A user can be an individual, system or application requiring access to Loom services or applications running on top of Loom.

  6. What can a user do?

    A user can place requests to Loom services such as job scheduling, logging, data connector management service and more. A user’s ability to access Loom services is under the control and responsibility of the Loom user account under which it is defined. You can permit a user to access any and all Loom services that have been integrated with IDM and to which the user account has subscribed. If permitted, a user has access to all the resources under that Loom account such as AssetDB (?), others.

Job Manager Service¶

Getting Started with Job Management Service¶

Loom Platform allows Loom Users to create various types of Policies that can help enterprises manage their data analytics as per business priorities and needs. There are two kinds of policies:

  • Loom Platform Policies
  • User Defined policies.

When a policy is published or a policy gets bind to specified enterprise information assets and a schedule is assigned to a policy, it results in the creation of a Loom Job Definition. Loom Job Scheduling framework uses these job definitions to create specific Job Instances that are based on the Job Definition Schedule. Job Management Service supports multi-tenancy. Each of the Loom Job is specific to a Loom Tenant.

Loom Dashboard has a Monitoring Menu on the sidebar. When a user clicks on this Monitoring Menu, based on the user role and privileges, the Jobs Page shows respective Jobs created for that specific user.

Overview¶

Version 0.6.0

Description Jobs management framework is a set of core platform services which help users to define policies which triggers Actions ( such as SCAN, DISCOVER, CLASSIFY, EXPORT, DELETE etc ). A Policy describes when the action will occur and scope of the action ( set of rules ). When a policy type is registered in the system, the corresponding Job Type is registered as well. There is one to one mapping between the policy type and Job Type.

For Each Job Type, there will be relevant platform service which will perform actual action based on Schedule.

1. What do Jobs mean from the Loom UI perspective? Loom application supports users to create various types of Policies. Loom will host Platform Policies and User Defined policies. When a policy is published (bind some assets and Assign schedule) it will create a Job Definition. These Job definitions will be accessed by Jobs Scheduling to create the Job Instances based on the Job Definition Schedule. The Jobs are tenant Specific. Based on the user role, tenant the respective Jobs are shown on Jobs Page under Monitoring Menu.

2. What is Jobs Monitoring? There is no jobs monitoring as such. The job definitions created after the Policy binding flow contain some schedule. So Loom scheduling services keeps watch on these schedules.

3. Is it any way linked with Health Monitoring? If yes, how? No. Health Monitoring focus on the Loom Infrastructure monitoring.

4. Who is eligible to view/ use this feature in the Loom UI? Platform Admin and Tenant Admins are eligible to See the Job list Feature.

5. Why only ____________ can view/ use this feature and not others? Tenant Admins: The jobs are specific to the tenants and they are executed on the data/infrastructure of the tenants. So Tenant Admin get to see all jobs cretaed for that tenant.

Platform Admins: There may be Loom jobs for Jobs DB, Audit Logs DB cleanups which are maintained by Platform Admins/IT Infrastructure Admins.

6. What does one achieve by using the Monitoring > Jobs feature? The loom user can monitor the status of Jobs created for various actions in the system. The scan, Discover are basic jobs but there may be specialized jobs e.g. classify, export where the output of Classify job will help user take further actions if he want to delete/archive the files. The actions based on jobs may vary.

Cluster Manager Service¶

Getting Started with Cluster Manager Service¶

Overview

Cluster Manager Service is one of the Loom Platform Service that runs in the Loom Control Plane. It is primarily responsible for managing and tracking Loom infrastructure components comprising of Control, Data Planes and Persistent Pod (PPOD). It has complete knowledge about the physical and logical deployment topology of the entire Loom setup. It can filter the topology information based on Loom Roles.

Version
0.6.0

Description

Cluster Manager provides API’s for creating, deleting and registering control and SaaS Data planes. This is also integrated with Loom UI.

Cluster Manager registers the cluster information first then invokes Airflow to install platform components. This also keeps tracks of installation status which can be viewed in the UI/API at runtime. Before triggering installation, it verifies availability of azure resources (Core and public IPs) for the selected configuration.

Tenant Admin Registers On-Prem data plane with CM from Loom UI and downloads starter file which is used to bring up On-Prem Setup. It also generates time bound azure URL for downloading OVA kit.

When tenant is on-boarded in IDM, CM receives that event, picks the appropriate Persistent pod for that tenant then triggers add tenant workflow.

This also helps to deploy security certificate on the nodes and enforces back up and restoration policy for all platform components.

Cluster Manager provides physical and logical topology of the entire platform. It also supports filtering of the topology based on role.

Cluster Manager Service: FAQ¶

General¶

What is Cluster Manager service?¶

Cluster manager service is one of the key Loom Platform Infrastructure Services that is responsible for managing and tracking Control, Data Planes and Persistent Pods of Loom Platform.

Service Dependencies¶

Cluster Manager Service depends on the overall health and functioning of the following Loom Core Platform Services:

  • Airflow is needed to run various Cluster Manager workflows.
  • PostgresDB is required for persisting cluster information.
  • KMS is for encrypting azure credentials.
  • Asset DB and cf-producer services are involved in Data Plane creation workflow.
  • Access to Azure API’s are needed to validate Azure resources and credentials.
What to use this service?¶

Platform admin uses Loom User Interface or REST APIs to create SaaS Control and Data planes. Tenant Admin role is responsible for registering On-Prem Data Plane by utlizing Cluster Manager Service.

Service Usage¶

Cluster Manager Service REST API workflow:

  • For SaaS Control and Data Planes deployment, Platform Admin enters node size, number of nodes and Azure credentials in Loom UI and then start the deployment process.
  • This results in a call being made to Cluster Manager which first verifies availability of Azure resources for the selected configuration and then persists the cluster details.
  • After that, it triggers Airflow to install Platform components and report the status back to the user.
  • For On-Premises deployment of Loom Platform and its Applications, the Tenant Admin registers Data Plane with Cluster Manager. Internally it makes a call to Asset DB and creates location and infrastructure assets.

Getting Started with Audit Logs Service¶

Loom platform consist of various services on both Control plane (CP) and Data plane (DP) layer. Some of the services are synchronous which provides rest endpoints for consumers. Other services are more of asynchronous in nature.

e.g. Loom UI triggers a Policy Binding flow to Policy-Admin-Service, but internally Policy-Admin-Service sends data to Policy-Service which send request to Jobs-Management Service to create Jobs for new policy. First part is synchronous while the other half is asynchronous.

Every action on the services, logs data using Audit-logs-services.

  1. Audit logs Dashboard: UI Dashboard present on the Loom Platform which enables users to view the Audit Logs for various services.
  2. Auditing Database: Responsible for storing all the log events generated by various services in the system
  3. Audit Logs Service: Rest Endpoint Services which provides way to fetch logs for the UI Dashboard.

Loom Asset Debuggability Guide¶

Loom Platform comprises of a bunch of micro-services that work together to provide data visibility, classification and discovery of enterprise data assets. Loom Asset Manager service and Loom Asset Database are two key services of Loom Platform that do most of the heavy-lifting when it comes to asset handling in a multi-tenant environment. While the Asset Manager service handles the function of Asset-Tenant mangement, who owns what assets, the Asset Database stores metadata about a tenant’s information assets and provides highly performant asset query APIs.

Figure: Loom Asset Database Service Overview

Loom Asset database provides a ‘near realtime’ reflection of information, applications and system’s inventory & status for the purposes of risk management, cost analysis and governance (info, security, business continuity, etc.). The Asset Database service doesn’t contain the actual assets that make up a company’s content repositories, but merely provides an inventory or catalog of the real-world state through one or more references (geo-locations, service naming, URL/URI, etc.).

The Loom Asset database comprises of Elasticsearch and Cassandra. In order to scale and isolate failures across tenants, Loom deploys multiple Elasticsearch and Cassandra clusters. Each such cluster is called a persistence pod. Each Persistence pod hosts a subset of tenants. A tenant data is always present inside one persistence pod.

This document is intended for Loom Developers and Loom Application Developers. The objective is to assist them in debugging issues related to Loom Asset Management and Loom AssetDb service. It describes tools and utilities designed to help with debugging Loom Asset Database and Asset Management service functionalities.

Overview¶

Asset Database is a complex service where data related to enterprise assets belonging to various tenents moves across several Loom components. At the time of diagnosing Loom Asset ingestion errors or failures, it can often take developers a lot of time to isolate the root cause of the issue. To help automate the process of checking various Loom components and pin point the actual cause of error, LoomPing utility can be utilized.

LoomPing Capabilities¶

LoomPing is under construction so some of the information in subsequent sections may be updated in the coming weeks.

Following is the list of key capabilties offered by LoomPing:

  • End to End validation of data ingestion pipeline - whether it is working fine of it there are any ‘data leaks’.
  • Tracker to help ensure data submission, delivery across the pipeline when data is sent in a controlled manner.
  • Report errors, if any, encountered during controlled data transfer and if there is a data loss in data ingestion pipeline.

How LoomPing Works?¶

LoomPing is implemented by deploying a diagnostic Pod on Loom Control Plane and another one on Loom Data Plane. It is a shell script which can be invoked as shown below.

loomping.sh <Data Plane Nginx IP> < TenantID > N
LoomPing Parameters¶
  • Data plane Nginx IP: Connection information about Diagnostic POD in data plane
  • TenantID: ID of the Tenant using which controlled messages will be generated to check the Asset Database Pipeline
  • N: Number of controlled diagnostic messages that will be generated
What happens when you run LoomPing?¶

When you run the loomping utility as shown in the command above, the following actions take place in Loom Diagnostic Pods deployed in the Data Plane and Control Plane:

Diagnostic Pod in Data Plane

  1. Produces scan and discover messages and put it on data plane Kafka.
  2. It also informs diagnostic POD on control plane about the number of messages generated for particular root-infra-asset under a TenantID.
  3. Gather status of activities from Control plane about each state
  4. Returns summarised report

Diagnostic Pod in Control Plane

  1. Monitors Kafka topics INFRASTRUCTURE_DISCOVERY and INFRASTRUCTURE_SCAN in control plane to verify that message pushed in DP are correctly received.
  2. Monitor Kakfka topic INFRASTRUCTURE_DISCOVERY_ASSETDB and INFRASTRUCTURE_SCAN_ASSETDB to ensure that the message consumer is processing the messages.
  3. Once the messages are validated in message consumer we validate persistence using following checks:
  • Verify that all the infra-assets and info-containers are stored in Janus
  • Verify that info-assets are stored in ElasticSearch and Cassandra
  • Verify that we can read messages back using object browser
